# План разработки

Документ описывает дорожную карту от MVP до зрелого продукта.

## Этап 0 — Инициализация (готово)
- ✅ Каркас Spring Boot + React, Docker-инфраструктура, CI-шаблоны.
- ✅ Базовые модели, миграции, security, REST + WebSocket.
- ✅ Демо-данные (admin/tourist/guide/hotel).

## Этап 1 — Закрытый MVP (2–3 недели)
**Цель:** проверить продукт на ограниченном количестве групп туристов.

1. Личные и групповые чаты (текст, файлы, прочтения) — **готово**.
2. Геолокация: точки сбора, текущая позиция — **готово**.
3. Каталог отелей и базовое бронирование — **готово**.
4. Push-уведомления через Web Push API.
5. Покрытие unit-тестами `ChatService`, `MessageService`, `BookingService`.
6. Логирование SLF4J + аудит входов в `audit_log` (новая миграция V4).
7. Минимальная админка: блокировка пользователя, просмотр жалоб.

## Этап 2 — Расширение домена (4–6 недель)
1. **Полноценный гид:** маршруты (множество waypoints), расписание на день,
   intent «найти ближайшую достопримечательность».
2. **Календарь номера** в отелях (overbooking-защита через `tsrange &&` индекс).
3. **Платёжная интеграция** с провайдером РФ (ЮKassa, Tinkoff Pay):
   - отдельный модуль `payments`, шаблонный сервис `PaymentGatewayClient`.
   - state-machine брони: PENDING → AWAITING_PAYMENT → CONFIRMED.
4. **Голосовые сообщения**: запись через MediaRecorder API, хранение opus в attachments.
5. **Поиск по сообщениям** — PostgreSQL FTS (`tsvector` колонка + `pg_trgm`).
6. **Модерация:** жалобы, теневое скрытие сообщений, фильтр запрещённого контента.

## Этап 3 — Подготовка к промышленной эксплуатации (4 недели)
1. Перевод сессий на Redis Cluster (для < 50ms задержки auth-проверок).
2. STOMP relay поверх RabbitMQ для масштабирования WebSocket > 1 узла.
3. Метрики Prometheus + Grafana (Spring Actuator → micrometer-registry-prometheus).
4. Трассировка OpenTelemetry → Jaeger (для запросов чат → сервис → БД).
5. Бэкапы PostgreSQL: `pg_basebackup` + `wal-g` в Yandex S3.
6. Сертификация ФЗ-152: документация по политике обработки ПДн, аудиты доступа,
   подача уведомления в Роскомнадзор.

## Этап 4 — Развитие
- Мобильные клиенты (React Native / Flutter) с тем же REST + WS API.
- Микросервисное разделение: `chat-service`, `booking-service`, `auth-service`, `media-service`
  — единый Postgres → схемы по сервису → отдельные БД.
- Видеозвонки (WebRTC + selective forwarding unit).
- AI-ассистент (рекомендации маршрутов, перевод сообщений с диалектов).

## Управление работой

- Доска: GitHub Projects (kanban TODO → DOING → REVIEW → DONE).
- Ветки: `main` (защищённая) ← PR из `feature/<TICKET>-...`.
- Релиз: semver + git-tag, образа в registry с тем же тегом.
- Definition of Done:
  - покрытие unit-тестами для нового публичного метода сервиса;
  - миграции с откатным `down` или явный `WARN: irreversible`;
  - обновлены `docs/api.md` и `docs/database.md` при изменении контракта;
  - локальный запуск `docker compose up` проходит smoke-сценарий.
